# 시공사례 관리자 + 목록 페이지 + 이미지 최적화 설계서

> 스택: React 18 + Vite + Tailwind + **Supabase**(Storage/DB/Auth) / 배포: **Vercel**
> 작성 기준: 현재 `src/App.tsx` 단일 파일, 라우팅·백엔드 없음, 이미지 하드코딩 상태

---

## 1. 목표 (요구사항 3가지)

1. **모바일형 관리자 페이지** — 메인 시공사례에 사진을 직접 업로드/삭제, 탭(카테고리) 분류
2. **시공사례 목록 페이지** — 캐러셀 탭별 "더보기" → 인스타 피드형(이미지 그리드) 목록, PC/모바일 반응형
3. **이미지 로딩 속도 개선** — 현재 첫 화면 수~십수 MB 로드 문제 해결

---

## 2. 아키텍처 개요

```
[관리자 휴대폰] --업로드--> [브라우저에서 WebP 압축/썸네일 생성]
                                   |
                                   v
                         [Supabase Storage] (원본 full + thumb)
                                   |
                         [Supabase DB: cases 테이블] (메타데이터)
                                   ^
        [메인 캐러셀 / 목록 페이지] --조회(SELECT)--
```

- **저장소**: Supabase Storage 버킷 `case-images` (public read)
- **DB**: Supabase Postgres `cases` 테이블
- **인증**: Supabase Auth (이메일/비밀번호) — 관리자 1~소수 계정
- **이미지 최적화**: **업로드 시 브라우저에서 압축**(무료 플랜 호환). Supabase 이미지 변환은 Pro 전용이므로 사용하지 않음.

## 3. Supabase 설정

### 3-1. Storage 버킷
- 버킷명: `case-images`, **Public** 체크
- 경로 규칙: `{category}/{uuid}-full.webp`, `{category}/{uuid}-thumb.webp`

### 3-2. DB 테이블 `cases`

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | uuid (PK, default `gen_random_uuid()`) | 사례 ID |
| `category` | text (`interior`/`special`/`europe`) | 탭 분류 |
| `full_url` | text | 큰 이미지 (목록/상세용, ~1600px WebP) |
| `thumb_url` | text | 썸네일 (피드/캐러셀용, ~600px WebP) |
| `title` | text (nullable) | 제목(선택) |
| `description` | text (nullable) | 설명(선택) |
| `sort_order` | int4 (default 0) | 정렬 우선순위(클수록 앞) |
| `created_at` | timestamptz (default `now()`) | 등록 시각 |

```sql
create table public.cases (
  id uuid primary key default gen_random_uuid(),
  category text not null check (category in ('interior','special','europe')),
  full_url text not null,
  thumb_url text not null,
  title text,
  description text,
  sort_order int4 not null default 0,
  created_at timestamptz not null default now()
);
alter table public.cases enable row level security;

-- 누구나 조회 가능(공개 사이트)
create policy "public read" on public.cases
  for select using (true);

-- 로그인한 관리자만 등록/수정/삭제
create policy "auth insert" on public.cases
  for insert to authenticated with check (true);
create policy "auth update" on public.cases
  for update to authenticated using (true);
create policy "auth delete" on public.cases
  for delete to authenticated using (true);
```

Storage 버킷 정책: SELECT는 public, INSERT/DELETE는 `authenticated`만.

### 3-3. 관리자 계정
- Supabase Dashboard → Authentication → Users → 관리자 이메일/비밀번호 수동 생성
- (회원가입 UI 불필요. 로그인만 구현)

## 5. 단계별 구현 계획

### Phase 0 — 이미지 최적화 (백엔드 무관, 즉시 효과)
가장 체감이 큰 부분. 기존 하드코딩 이미지에도 바로 적용.
1. `KVimg.png`(4MB) 등 PNG 사진 → **WebP 변환 + 리사이즈** (빌드 전 1회 일괄 변환 스크립트 또는 수동)
2. 모든 `<img>`에 `loading="lazy"` `decoding="async"` 추가 (단, KV 메인 이미지는 `eager`)
3. 캐러셀/그리드 이미지에 썸네일 사용, 클릭 시 원본
4. `index.html`의 `<meta viewport width=1920>` → `width=device-width` 로 교정 (현재 모바일에서 강제 1920 고정 → 반응형 정상화)

> 예상 효과: 첫 화면 전송량 70~85% 감소, LCP 대폭 개선.

### Phase 1 — 라우팅 + Supabase 연결
1. `npm i react-router-dom @supabase/supabase-js`
2. `src/lib/supabase.ts` — `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` 사용
3. `main.tsx`에 `<BrowserRouter>`, `App.tsx`에 `/`, `/cases`, `/cases/:category`, `/admin` 라우트
4. `vercel.json`에 SPA rewrite 추가(아래 7장)

### Phase 2 — 데이터 연동 (메인 캐러셀)
1. `cases.ts`의 `fetchByCategory(category)` 작성
2. 기존 `Section1` 캐러셀이 하드코딩 배열 대신 Supabase에서 조회하도록 변경
3. **기존 이미지 시드**: 현재 `/public/images`의 사례 이미지를 관리자 페이지(또는 1회 스크립트)로 업로드해 데이터 이관
4. 각 탭 위/아래에 **"더보기"** 버튼 → `/cases/{category}` 이동

### Phase 3 — 시공사례 목록 페이지 (인스타 피드)
1. `CasesPage` — 상단 카테고리 탭(전체/인테리어/스페셜/유럽미장)
2. `CaseGrid` — 반응형 그리드: **모바일 2열 → 태블릿 3열 → PC 4열** (CSS grid 또는 masonry)
3. 무한 스크롤 또는 "더 보기" 페이지네이션 (range 쿼리)
4. 카드 클릭 → 라이트박스(확대 보기) — 선택 사항

### Phase 4 — 모바일 관리자 페이지
1. `/admin` 진입 시 미로그인 → 로그인 폼 (Supabase `signInWithPassword`)
2. 로그인 후:
   - **업로드 폼**: 카테고리 선택 + 사진 다중 선택(`<input type=file accept="image/*" multiple>`, 모바일에서 카메라/갤러리 연동) + 제목/설명(선택)
   - 업로드 시 `imageCompress.ts`로 full(1600px)/thumb(600px) WebP 2종 생성 → Storage 업로드 → `cases` insert
   - **사례 목록**: 등록된 사례 썸네일 그리드 + 삭제 버튼(Storage 파일 + DB row 동시 삭제) + 정렬 변경
3. 모바일 우선 레이아웃 (큰 터치 버튼, 풀폭 입력)

### Phase 5 — 배포
- Vercel 환경변수 등록, `vercel.json` rewrite 확인, 빌드/배포 검증

## 6. 이미지 압축 유틸 설계 (`imageCompress.ts`)

```ts
// 브라우저 Canvas 기반. 외부 라이브러리 없이 구현 가능.
// 입력: File, 최대 너비, 품질 → 출력: WebP Blob
export async function toWebp(file: File, maxW: number, quality = 0.82): Promise<Blob> {
  const img = await loadImage(file);          // createImageBitmap 또는 Image
  const scale = Math.min(1, maxW / img.width);
  const canvas = document.createElement('canvas');
  canvas.width = Math.round(img.width * scale);
  canvas.height = Math.round(img.height * scale);
  canvas.getContext('2d')!.drawImage(img, 0, 0, canvas.width, canvas.height);
  return await new Promise(res => canvas.toBlob(b => res(b!), 'image/webp', quality));
}
// 업로드 시: full = toWebp(file, 1600), thumb = toWebp(file, 600)
```

> 라이브러리를 쓰고 싶다면 `browser-image-compression` 권장. 무외부 의존 방식도 충분히 동작.

## 7. Vercel 배포 설정

SPA 클라이언트 라우팅(`/cases`, `/admin` 직접 접근/새로고침) 대응을 위해 rewrite 필요:

```json
// vercel.json
{
  "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
}
```

환경변수 (Vercel Project Settings → Environment Variables):
- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

> anon key는 공개되어도 안전(RLS로 보호). **service_role 키는 절대 프론트에 넣지 않음.**

## 8. 반응형/모바일 그리드 기준

| 화면폭 | 목록 그리드 | 캐러셀 |
|---|---|---|
| < 640px (모바일) | 2열 | 1.x장 보이는 스와이프(현행 유지) |
| 640~1024px (태블릿) | 3열 | 3장 |
| > 1024px (PC) | 4열 | 6장(현행) |

- `index.html` viewport를 `width=device-width`로 교정해야 모바일 반응형이 정상 동작 (현재 `width=1920` 강제 고정 상태).
